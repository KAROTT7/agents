# 设计 Token 与视觉规范输出

把设计方案沉淀成 token 和规范的目标，是让界面能够被稳定复用，而不是停留在一次性设计描述。

## 什么时候需要输出 token

- 页面或组件会持续扩展，不只做一次性设计。
- 前端需要统一变量、主题或组件库配置。
- 多个页面已经出现颜色、间距、圆角、阴影不一致的问题。
- 需要把视觉方向转成设计系统或开发规范。

## 最小 token 集

### 颜色

- 品牌主色
- 品牌辅助色
- 中性色阶
- 成功、警告、错误、信息状态色
- 背景层级色
- 边框色
- 文字主次色

### 字体

- 字体家族
- 标题字重
- 正文字重
- 字号层级
- 行高层级
- 字间距规则

### 间距

- 基础单位
- 组件内边距
- 区块间距
- 栅格 gutter
- 页面边距

### 形状与层次

- 圆角层级
- 边框粗细
- 阴影层级
- 模糊和叠层强度

## 命名原则

- 用稳定、可扩展的语义命名，不直接绑死某个具体页面。
- 优先使用用途命名，例如 `color-text-primary`，而不是 `blue-500-for-homepage`。
- 区分基础 token 和语义 token。
- 不要把一次性视觉效果混进全局 token。

## 基础 token 与语义 token

### 基础 token

- 用来定义原始颜色值、字号值、间距值、圆角值。
- 适合作为设计和代码中的底层值来源。

### 语义 token

- 用来表达用途，例如主文字、卡片背景、主按钮背景、危险操作边框。
- 适合直接映射到组件和状态。

## 组件规范最少写什么

- 组件名称与用途
- 结构组成
- 默认尺寸与可选尺寸
- 内外间距规则
- 各状态表现
- 是否支持图标、标签、说明文案
- 禁止使用场景

## 状态规范最少写什么

- 默认态
- 悬停态
- 激活态
- 聚焦态
- 禁用态
- 加载态
- 错误态
- 成功态

## 页面级规范最少写什么

- 栅格系统
- 页面边距
- 区块间距
- 标题层级
- 卡片和容器规则
- 深浅背景切换规则
- 桌面端与移动端重排原则

## 输出格式建议

- 先给视觉方向摘要，说明风格目标。
- 再给 token 分组：颜色、字体、间距、圆角、阴影。
- 再给组件规范表或组件说明。
- 最后给页面使用规则和例外说明。

## 常见问题

- 只有设计稿，没有可复用变量。
- token 名称过度绑定当前业务，后续难扩展。
- 状态色和品牌色混用，导致语义不清。
- 组件规范缺少禁用态、错误态和边界条件。
- 规范写得太抽象，开发无法直接落地。

## 输出示例骨架

- 视觉摘要：2 到 4 句说明整体风格和产品目标。
- Token 列表：颜色、字体、间距、圆角、阴影。
- 组件列表：按钮、输入框、卡片、表格、弹层、导航。
- 页面规则：布局、区块、响应式、状态反馈。
